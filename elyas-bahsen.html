Dependability · Security · Availability
TANDIB · Group 2 · 16 June 2026

The Concepts I Implemented

My responsibility was to make TANDIB a dependable system that people can trust — focusing on the security and availability attributes of dependability from Sommerville Ch.10–14.

Dependability — the guiding goal

I applied dependability as the overall design principle: a system is only useful if people can trust it. I made "the system must protect its users and keep working" a requirement from the start, not something added later, so that trust is built into how TANDIB is designed.

How I implemented Security

  • I implemented secure identification, so the system always knows who is making a request before it responds.
  • I enforced that each user can only ever reach their own information and never another person's — confidentiality by design.
  • I applied the principle of least privilege: every type of user is granted only the permissions their role genuinely needs.
  • I followed the rule of never trusting the user's side — sensitive decisions are confirmed on the trusted side so they cannot be bypassed.

How I implemented Availability

  • I made the system run continuously and reachable at any time, so it behaves like a real always-on service rather than something that only runs on demand.
  • I designed it so it can serve many users at once and grow without being rebuilt, supporting availability as the user base increases.

Reflection

The most important thing I learned is that dependability and security cannot be bolted on at the end — they must shape the design from the beginning. By treating "protect every user" and "always be available" as core requirements, the whole project became something people can actually trust, which is the real difference between writing software and engineering it.
