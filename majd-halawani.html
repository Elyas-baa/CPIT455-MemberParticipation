Software Reuse · Component-Based Software Engineering
TANDIB · Group 2 · 16 June 2026

The Concepts I Implemented

My responsibility was to apply software reuse and component-based software engineering (Sommerville Ch.15–16) so TANDIB is built efficiently from trusted, reusable parts rather than written from scratch.

How I implemented Software Reuse

  • Instead of writing every capability from zero, I built on proven, well-tested building blocks, so the project inherits their reliability and security.
  • I applied reuse to reduce both the effort and the number of defects — reused parts have already been used and tested by many others.
  • I created shared, common elements once and reused them across the whole product, so the system stays consistent and changes happen in one place.
  • This let our small team deliver far more, and more reliably, than if we had hand-built everything.

How I implemented Component-Based SE

  • I designed the system as a set of self-contained components, each with one clear responsibility and a clear boundary.
  • Because each component is independent, one can be improved or replaced without breaking the others.
  • This composition made the system easy to assemble: complete parts are put together like building blocks instead of being tangled into one mass of code.
  • It also enabled real teamwork — each member could work on a different component at the same time without conflicts.

Reflection

I learned that good engineering is often about not rebuilding what already exists. Reusing trusted components gave us quality we could never have written ourselves in the time available, and designing the system as independent components is what made it possible for four people to work as one team. These two ideas together — reuse and components — are what kept the project both fast and dependable.
